# scanvoca-feature — 기능 개발 오케스트레이터

**실행 모드: 서브 에이전트** (이 환경에는 TeamCreate가 없다. `Agent` 도구로 스폰하고, 후속 지시는 `SendMessage`로 같은 에이전트를 이어간다. 모든 Agent 호출에 `model: "opus"`를 명시한다.)

## 규모 판별 (에이전트 스폰 전 필수)

에이전트 스폰은 비용이 크다. 먼저 작업 규모를 판별한다:

| 규모 | 기준 | 실행 방식 |
|------|------|----------|
| 소규모 | 파일 1~3개, 단일 영역(FE만 또는 BE만) | **메인이 직접 구현** — 해당 영역 스킬(scanvoca-frontend/backend)을 읽고 작업, QA 체크리스트(scanvoca-qa)도 메인이 직접 수행 |
| 중규모 이상 | FE+BE를 모두 관통하거나, 독립 모듈 여러 개 | 아래 Phase 워크플로우로 에이전트 분업 |

## Phase 0: 컨텍스트 확인

1. `_workspace/` 존재 여부 확인:
   - 존재 + 부분 수정 요청 → **부분 재실행**: 해당 에이전트만 재호출(SendMessage 우선), 이전 산출물 경로를 입력으로 전달
   - 존재 + 새 기능 요청 → **새 실행**: 기존 `_workspace/`를 `_workspace_prev/`로 이동 후 시작
   - 미존재 → **초기 실행**
2. `git status`로 작업 트리 상태 확인 — 미커밋 변경이 있으면 사용자에게 알린다.

## Phase 1: 분석 및 계약 정의 (메인 직접)

1. 요구사항을 FE 작업 / BE 작업 / 공통(타입·계약)으로 분해한다.
2. FE·BE 양쪽에 걸치면 **API 계약을 먼저 확정**하여 `_workspace/01_contract.md`에 기록한다:
   - 엔드포인트 경로·메서드, 요청 body, 응답 스키마(필드명·타입·Optional), 에러 케이스(`detail` 문자열)
3. 계약이 확정되어야 FE·BE를 병렬로 진행할 수 있다.

## Phase 2: 구현 (서브 에이전트 병렬)

```
Agent(subagent_type: "general-purpose", model: "opus", run_in_background: true)
  ├── be-dev 정의(.claude/agents/be-dev.md)를 프롬프트에 포함 + 계약 파일 경로 전달
  └── fe-dev 정의(.claude/agents/fe-dev.md)를 프롬프트에 포함 + 계약 파일 경로 전달
```
- FE·BE가 독립적이면 병렬 실행, 의존이 있으면 BE(계약 구현) → FE 순서로 실행한다.
- 각 에이전트의 완료 보고(변경 파일 + 검증 결과)를 수집한다. 검증 미통과 보고는 완료로 취급하지 않는다.

## Phase 3: QA (서브 에이전트 또는 메인 직접)

- qa 에이전트 정의(`.claude/agents/qa.md`)와 `scanvoca-qa` 스킬로 경계면 교차 비교 + 정적 검증 + 스모크를 수행한다.
- FAIL 발생 시: 해당 개발 에이전트에게 `SendMessage`로 QA 리포트를 전달해 수정시키고, 수정 후 qa에게 재검증을 요청한다. **수정→재검증 루프는 최대 2회**, 그래도 FAIL이면 사용자에게 상황을 보고하고 지시를 기다린다.

## Phase 4: 배포 (메인 직접)

- `scanvoca-deploy` 스킬의 절차를 따른다.
- 사용자가 요청에 "배포까지"를 명시하지 않았다면 QA 통과 시점에 배포 여부를 확인받는다.

## 데이터 전달 프로토콜

- **계약/산출물**: 파일 기반 — `_workspace/{순번}_{에이전트}_{산출물}.md` (예: `01_contract.md`, `02_fe_changes.md`, `03_qa_report.md`)
- **결과 수집**: Agent 반환 메시지
- **수정 지시**: SendMessage (같은 에이전트의 컨텍스트 유지)
- `_workspace/`는 커밋하지 않는다 (.gitignore 확인).

## 에러 핸들링

| 상황 | 대응 |
|------|------|
| 에이전트 검증 실패 보고 | 에러 로그와 함께 1회 재지시, 재실패 시 사용자 보고 |
| FE-BE 계약 불일치 발견 | 계약 파일을 기준으로 판정 — 계약과 다른 쪽이 수정. 계약 자체가 틀렸으면 계약 갱신 후 양쪽에 통지 |
| 배포 실패 | 배포 로그 확인 후 원인 보고. 재배포는 원인 수정 후에만 |
| 운영 DB 파괴 위험 감지 | 즉시 중단하고 사용자 확인 요청 |

## 테스트 시나리오

**정상 흐름**: "단어장에 즐겨찾기 기능 추가해줘" → Phase 1에서 계약 정의(PATCH /wordbooks/{id} + is_favorite 필드) → be-dev(모델+마이그레이션+테스트)와 fe-dev(토글 UI) 병렬 → qa가 스키마↔타입 대조 + pytest + build 확인 → 배포 여부 확인 후 push.

**에러 흐름**: qa가 "응답 필드는 is_favorite인데 프론트 타입은 isFavorite" FAIL 보고 → 계약 파일 기준(snake_case)으로 fe-dev에게 SendMessage 수정 지시 → 재검증 PASS → 진행.

**소규모 흐름**: "설정 페이지 버튼 색만 바꿔줘" → 규모 판별에서 소규모 → 에이전트 스폰 없이 메인이 scanvoca-frontend 스킬 읽고 직접 수정 → tsc/build 확인 → 완료.
